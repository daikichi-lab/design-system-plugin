# Visual Psychology — 視線設計の原則（emphasis / peak / 顕著性）

「崩れない・読める」（typography / kinsoku / whitespace / 高さゲートの床）の
**上に載せる層**：視線がどこから入り、どこで止まるかの設計。全要素が等価に
並ぶスライドは「どこも見なくていい」スライドと同じ——退屈さは情報設計の欠陥
である。ただしこの層は自由な装飾ではなく、**トークン＋機構＋検証**で運用する
（機構：`emphasis` スロット / `peak` 属性、検証：design-lint の
EMPHASIS-COUNT / SALIENCY / RHYTHM）。`deck-strategy`（主役と山場を決める）と
`deck-review`（強調の誠実さを確認する）が読む判断原則がこの文書。

> 強調の最大の失敗モードは**乱用**である。全部を強調すれば何も強調されない。
> だから「1スライド1強調・デッキ1山場」は好みではなく機械が守る制約
> （EMPHASIS-COUNT は ERROR）。

---

## 1. 視覚階層 — 主役は1つ、序列は面積で作る

人は最も大きい・濃い・彩度の高いものを最初に見る——**そして色より面積が強い**。
初回A/Bの実測が証明した：stat-grid の淡ティント＋小さなサイズステップでは、
主役「8.4%」が文字数の多い脇役「48.2億円」に負けた（レンダ画像の顕著性計測
でも 55 vs 69 で敗北）。色の淡い変化だけでは面積の序列を逆転できない。

1スライドの主役は1つ（one slide / one message の視覚版）。主役が言えない
スライドは、メッセージが曖昧な兆候——ビートを見直す（deck-strategy Step 3）。

- 機構：`content.emphasis`（要素 index、パターンごとに1つだけ）。
- **stat-grid は面積の序列**：主役カード幅 ×1.45（残りは等分で細く・合計幅
  不変）＋主役の数字 ×1.7（peak では ×1.8）＋単位は小さく添える＋
  accentDeep／ティント。脇役の数字はサイズ現状維持（単位のみ小さく揃える）。
- **数値＋単位は非分割アトム（層①・機械規則）**：518億円 は 518/億円 にも
  億/円 にも割れない（装飾 約・＋・▲・カンマも同一アトム）。あふれは
  ①アトム全体の等比縮小（可読フロア：脇=基準×0.7≒主役字高の41%、
  主役=基準×1.6）→②カード側が幅を譲る（per-card 最小拡幅・主役幅を控える、
  発火はログ）→③それでも不可なら design-lint ERROR（内容を短くする）。
  **アトムを割る折返しには決して戻らない。** 散文（takeaway 等）でも同じ
  規則を bake の budoux 境界に接木（measure.js の UNIT_ATOM_RE）。
- flow / cycle は accent 塗り＋onDark＋×1.15（塗りの反転が面積級に強いため
  これで足りる——A/B で主役が明確に立った）。
- 実装は既存トークンのみ。新しい色・縁取り・ストライプは発明しない
  （house-quality-bar §2）。定数は diagrams.js（EMPH_CARD_SCALE /
  VALUE_JUMP / EMPH_SCALE）が単一ソース：エンジンの描画と床の計測が
  同じ数値を使う。

## 2. フォン・レストルフ効果 — 異質は希少でこそ効く

均質な列の中の1つの異質が記憶に残る（isolation effect）。逆に言えば、
異質が2つあれば効果は割れ、全てが異質なら消える。emphasis が
**色＋面積のシングルトン**として実装されているのはこのため。

**マーカー（`content.marker`）— 序列の上に「見せ場」を作る装置。** 色・太さの
序列は視線の順番を作るが、それだけでは退屈が残る——作者の印は
「ここを見よ」という信号になる。いずれも主役と同一要素にのみ：

- `badge`（**既定**） — 主役の肩に乗る小さな塗りピル＋**ネイティブ文字**の
  事実ラベル（「過去最高」「初」「3期連続」——データが支持する事実のみ、
  ≤8字）。形＋ラベル＋位置の冗長チャネル＝色覚に頑健・数字に重ならない・
  on-register（第三者レビューで最良評価）。迷ったら badge。
- `underline` — 主役の数字の下に置く**ソリッドな下線バー**（意図的な幾何、
  ネイティブ形状）。より強いポップが要るときの「大人の語彙」。
- `arrow-note` — 主役に**接続する**短い矢印＋一言注釈（ネイティブ文字、
  ≤14字）。注釈は**カードに既出でない情報のみ**——既出文言の重複は装飾
  （design-lint が ERROR で弾く）。
- `circle`（**register ゲート**） — 手描き風の囲みは「採点マーク・落書き注釈」
  の語彙で、**財務・ビジネス register と衝突する**（第三者レビューの確定
  事実）。テーマが `layout.marker.handDrawn: true` を明示した場合のみ選択
  可能——**同梱テーマは全て閉鎖**（editorial 系プロジェクトが意図して開ける
  余地だけ残す）。手描き揺れの装飾は当該 register では不可、が原則。

強い装置ほど乱用の害も大きい：**マーカーも1スライド1つ**（構造上
`marker` は1つしか書けず、主役以外には付かない）。検証：EMPHASIS-COUNT
（`emphasis` の重複指定・legacy `emphasizeIndex` との併記・マーカーの
型/対象/文言違反・circle の register 違反・注釈の重複 = ERROR）。
SALIENCY リント（§下表）は「主役より目立つ脇役」をレンダ画像で検出する
（§7 の限界を必ず読むこと）。

## 2.5 色覚・強調の床規則（層①・常時ON — Color Universal Design）

今回の実験群が示したのは「濃淡 < 反対色」ではなく**「色 < 面積」**である。
勝った梃子は面積（実測 1.70×）。この事実の上に、常時ONの床規則を置く：

- **強調は「面積・サイズ・位置・形/ラベル」の冗長多重で担い、色相は補助。
  色相を唯一の担い手にしない。** 色覚特性は約12人に1人——色相単独の強調は
  約8%の読者で崩れる。エンジンの標準治療はこの床を満たす（stat-grid=面積
  ×1.45＋サイズ×1.7、flow/cycle=塗りの明度反転＋サイズ、badge=形＋ラベル
  ＋位置）。legacy `emphasizeIndex`（ティントのみ・サイズ/面積差ゼロ）は
  design-lint が SALIENCY WARN で移行を促す。
- **反対色・警告色は財務物で二重に危険**：ブランド（抑制＝信頼が壊れる）と
  財務セマンティクス（赤橙＝損失・危険——良い数字に載せると意味が逆走し
  誘導リスク）。**警告色による強調は不可**。強調は accent 系トークンの
  範囲でのみ行う（本プラグインの機構はトークン外の色を持たないので、
  プロジェクトが theme の accent に警告色を入れない限り守られる——
  theme-init / design-doc の責務）。
- **財務・取締役会系 register は既定で抑制**：強調は任意（「強調しない」が
  正しい結論であることも多い——抑制＝信頼）。使うなら面積スキームか badge
  のみ。手描き語彙は不可（§2 の circle ゲート）。

## 3. ゲシュタルト（近接・類同・囲み） — 間隔は情報

意味的に近いものは物理的に近く。カードの gap、行間、余白の均一性は
whitespace.md の床が守る。この層で足すのは「**違うものは違って見せる**」
（類同の破れ = emphasis）だけで、グルーピング自体は床の仕事。

## 3.5 記号は「評価」を運ぶ — 文字記号優先の決定木

テキストが**内容**を運ぶのに対し、記号は**聴衆が各項目に下すべき評価**を
先回りして宣言する装置である。置いてよいのはその仕事があるときだけ：

- **✕**（痛み色）＝「これは否定です」— 読者が「ダメな例だ」と判断する手間を省く
- **数字丸**＝「順番があります」
- **→**＝「変換・因果です」
- **＝**＝「同じものです」

逆に、評価を運ばない**絵アイコン（電球・ロケット・握手の類）は情報量ゼロの
装飾**＝AIっぽさの典型（house-quality-bar §2 の clip-art / emoji 系列）。
決定木は一つ：**文字記号（✕ → ＝ 数字）で足りるなら絵を使わない。絵を
使ってよいのは概念そのものが空間的なときだけ**——それは図解スケルトン
（flow / cycle / matrix …）の領分であり、その場合も「言葉を図形に格納する」
のであって挿絵を描くのではない。パターンはこの写像を既に内蔵している
（numCircle=順序、flow の矢印、formula の演算子、dialogue の○×ラベル）——
新しい記号を発明しない。0.5秒テスト（文字をグレー線に置換しても項目数・
答え・各項目の○✕が伝わるか — `slide-design-principles.md` §7）が検算になる。

**図解の定義と「額装」の禁止（記号消去テスト）。** 図解とは「面積・長さ・
位置が意味を運んでいる」状態を指す。文を箱に入れて ＝ ＋ − → でつなぐだけの
構図は図解ではなく**文章の額装**であり、認知負荷はテキストと変わらない。
検算：**記号を全部消しても関係が伝わるか**。伝わらなければ、量を寸法に・
関係を位置に翻訳し直す（同サイズの箱を横に並べて記号でつなぐ構図は原則
額装）。図解の形は原稿から導出できない——その分野の**正準形**（実務で使われる
図の型：BS=左右の積み上げ恒等式（`identity` 骨格）、資金の流れ=waterfall、規模比較=実寸横棒（chart `bar`）、
決算書イメージ=書類風ミニチュア、構成比=帯）を呼び出す。正準形が分からない
概念は、実装前に調査するか人間に確認する（diagram-recipes.md の各骨格が
このライブラリの機械化面）。

## 4. 視線動線 — 主役は視線の入口〜第一停留点に

読者は左上から入る（Z / F パターン）。kicker → title → 本文という既定の
動線の中で、emphasis は第一停留点を**作者が指定する**手段である。
チャートの emphasizeIndex（1本だけ accentDeep、他は accentSoft）も同じ機構：
色が視線を導き、データラベルは全バーに残る（色は導くが、データは隠さない）。

## 5. 緩急とピーク・エンドの法則 — デッキに山場を1つ

記憶に残るのは**山場（peak）と最後（end）**。だから：

- **`peak: true`（slide レベル・デッキ最大1枚・body のみ）**：ONE main
  message を証明するスライド。視覚コントラストの上限を一段開放する——
  message は surfaceAccent の地 + stat 一段大きく（accentDeep）、
  emphasis 対応パターンでは強調のサイズステップが ×1.15 → ×1.3 に開く
  （PEAK_EMPH_SCALE）。**peak 以外は従来の抑制を保つ**——緩急はこの対比から
  生まれる。天井を毎枚開ければ、それは新しい床にすぎない。
- **end = cta**：デッキは必ず単一アクションの cta で終わる（既存の砂 sandwich
  規則）。山場 + cta が記憶に残る2点になるよう配置する（deck-strategy Step 4）。
- 検証：peak 2枚以上 = ERROR、cover/cta への peak = ERROR、
  RHYTHM（密なスライド3連続以上 = WARN、advisory）。

## 6. アンカリング・系列位置効果 — 数字の置き場所

最初に見せる数字が基準（アンカー）になる。重要な数字・項目は**先頭か末尾**
（初頭効果・新近効果）、中央に埋めない。stat-grid の主役 KPI は端のカードに、
表の結論行は最初か最後に。これはリントでは守れない——deck-strategy の
判断であり、notes に根拠を残す。

## 7. 誠実ガード（MUST — 機械では守れない領域）

**強調してよいのは、データが実際に支持する結論だけ。強い装置ほど厳格に。**

- 都合の良い数字だけを大きくする、損失回避の煽り言葉で誘導する、
  会社予想・推定値を実績と同じ顔で強調する——いずれも
  house-quality-bar §4 違反。誘惑と誠実の境界を越えない。
- **badge / arrow-note の文言は事実のみ**：「過去最高」は本当に過去最高の
  時だけ。誇張形容（驚異・圧倒的・衝撃・奇跡 等）は design-lint が機械的に
  ブロックするが、**文言の事実性そのもの**（本当に「初」か・本当に
  「3期連続」か）は deck-review の人の承認領域。
- 実例（サンリオ deck）：営業利益の系列で強調するのは**実績の過去最高
  （518億円）**であり、会社予想（751億円）ではない。予想はミュートのまま、
  注記で「会社予想」と明示する。このとき SALIENCY は「予想のバー（物理的に
  最大）が主役より目立つ」と advisory WARN を出す——**正しい検出であり、
  誠実ガードを優先して WARN を受容する**（理由を notes に記録）。強調を
  予想に移して WARN を消すのは本末転倒。
- 財務・事実系のデッキでは特に、emphasis / peak / marker が付いた数字・
  主張を deck-review が**重点確認**する（機械が守るのは「乱用の禁止」まで、
  **意味の正しさは人の承認領域**）。

## 機構と検証の対応表

| 原則 | 機構（deck plan） | 検証 |
|---|---|---|
| 視覚階層 / フォン・レストルフ | `content.emphasis`（1スライド1つ；stat-grid は面積の序列） | design-lint EMPHASIS-COUNT（重複・範囲外・主役値の幅超過 = ERROR） |
| 見せ場（作者の指差し） | `content.marker`（circle / badge / arrow-note、主役のみ） | design-lint（型・対象・文言・誇張語 = ERROR）＋ image-lint MARKER（中心透明・解像度） |
| 主役が主役に見えるか | — | **saliency-lint（レンダ画像：ink面積×コントラスト×彩度、argmax 判定、WARN）** |
| ピーク・エンド | `slide.peak`（デッキ1枚・body のみ） | peak ≥2 / cover・cta = ERROR |
| 緩急 | パターンの並び（deck-strategy Step 4） | RHYTHM（密3連続以上 = WARN・advisory） |
| 誠実ガード | notes に強調・マーカーの根拠を記録 | deck-review の重点確認（人） |

パターン別の emphasis / marker の効き方は `references/patterns/catalog.md` と
`references/graphics/diagram-recipes.md` の各ブロック。実装は
`bin/generate.js`（描画）＋ `bin/layout-html/geometry.js`（床が強調サイズ・
非対称幅で測る）＋ `bin/graphics/diagrams.js`（EMPH_SCALE / EMPH_CARD_SCALE /
VALUE_JUMP / statGridLayout の単一定義）＋ `bin/graphics/recipes.js` &
`make-markers.js`（circle マーカー資産）＋ `bin/lint/saliency-lint.js`
（build.sh がレンダ後に実行）。

## 正直な限界（この層が保証しないこと）

- **SALIENCY はレンダ画像ベースの近似**（ink 面積×コントラスト×彩度の積分、
  argmax 判定）であり、実際の視線を保証しない。受け入れ基準は「初回A/Bの
  失敗（淡ティントの主役が長い脇役に負けた実例）を WARN できること」で、
  その実例に対して較正済み。視線予測ではない。最終判断は目（M-2）。
  静的トークン近似は当の失敗を見逃したため**廃止**した。
- **強調・マーカー文言の意味的な正しさ**（データが支持するか・誘導的で
  ないか）は機械で守れない——deck-review の人の承認領域（§7）。
- 緩急・階層は原理であって公式ではない。RHYTHM / SALIENCY は advisory 中心
  （誠実ガード優先の意図的 WARN 受容もある——§7 の 751 例）。
- 強調のサイズステップ・幅非対称はラベル/値の字数予算を削る——床（bake +
  高さゲート）と主役値の幅ゲート・ラベル幅 WARN が超過を検出する
  （強調するなら短く言い切る；単位は値の小添えか label へ）。
- `peak` の視覚効果が定義されているのは message（地 + stat）と emphasis 対応
  パターン（ステップ開放）のみ。それ以外のパターンでは peak は戦略マーカー
  （リント対象）であって見た目は変わらない。
- 字面は実機依存（既存の性質）：saliency-lint は soffice レンダを測る。実機
  PowerPoint の字形差で僅かに数値は動く（argmax 判定は概ね保たれる想定）。
